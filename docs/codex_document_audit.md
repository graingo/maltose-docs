# Maltose 文档审计报告

## 说明

本文档基于本地仓库以下目录的对照阅读整理：

- `maltose-docs`
- `maltose`

关注点主要包括：

- 文档设计是否清晰
- 文案表达是否成熟
- 文档内容是否与当前代码实现一致

本文档不讨论品牌包装或宣传文案，只聚焦文档本身的工程质量与可用性。

## 问题

### 1. 存在整章内容与代码实现脱节的情况

最明显的是并发控制章节。文档声称框架提供 `util/msync`、`SingleFlight`、`LockedCalls`、`Limit`、`Pool` 等能力，但当前 `maltose` 仓库中并不存在对应模块。这个问题不是措辞误差，而是功能层面的不对应。

影响：

- 用户会误以为框架已内置并发控制工具
- 文档会把不存在的能力包装成正式特性
- 读者很容易在实际使用时产生落差

### 2. 一部分基础示例已经不能代表当前公开 API

`guide` 和 `components/server` 中多处示例仍沿用旧风格或假想 API，例如：

- 使用 `r.Response.Write(...)`
- 使用 `mhttp.Context`
- 使用 `TraceIDFromContext`
- 使用 `AbortWithError`
- 使用 `SetCtxVar` / `GetCtxVar`
- 使用 `MiddlewareCORS`

这些名称和当前代码中的公开接口并不一致，部分甚至在仓库里根本不存在。对读者来说，这类问题比概念性偏差更严重，因为会直接导致示例无法照抄、无法编译、无法运行。

### 3. 配置文档与运行时读取逻辑不一致

日志章节和完整配置参考中，顶层日志配置写成了 `log:`，但当前框架运行时读取的是 `logger` 节点。类似问题还包括字段命名和上下文字段注入说明。

这类问题的风险很高，因为用户通常会优先复制文档中的配置片段。只要配置键名写错，用户就会得到“代码没生效、但又看不出哪里错”的体验。

### 4. 组件包装后的调用方式，没有在文档中体现清楚

Redis 文档中大量示例按 go-redis 原生链式结果写法来展示，例如 `Set(...).Err()`、`Get(...).Result()`，但当前 `mredis` 的封装返回值并不是这个接口风格。

同类问题也出现在部分 Web 示例中，文档展示的是一个“想象中的上层体验”，但代码里实际暴露的是另一套调用方式。

结果是：

- 文档看起来顺手
- 用户真实调用时却发现签名不一样
- 读者会怀疑是自己用错了，而不是文档过期

### 5. 路由与 OpenAPI 标签说明存在不一致

路由文档将 `m.Meta` 的标签写成 `tags:"..."`，但当前代码解析使用的是 `tag`。这会造成两层误导：

- Web 运行时自动文档生成拿不到标签
- CLI 生成的 OpenAPI 文档分组与预期不一致

这类问题不容易在第一时间暴露，但一旦用户开始生成文档，就会发现“写了标签却没生效”。

### 6. 错误处理章节对 HTTP 状态码的描述与实现相反

文档中提到使用 `MiddlewareResponse()` 后，业务错误通常仍然返回 `200 OK`。但当前实现实际会把常见错误码映射到 `400/401/403/404/500` 等 HTTP 状态码。

这个问题会直接影响前后端联调预期，也会影响调用方如何设计错误处理逻辑。

### 7. FAQ 中仍保留过时技术叙述

例如 FAQ 中提到日志组件选型时写到了 Logrus，但当前代码和其他文档主体已经转向 Zap 风格描述。这说明 FAQ 没有跟随主文档和代码一起更新。

FAQ 的问题不在于篇幅短，而在于它通常被读者视为“高密度结论区”。一旦 FAQ 过时，就会比正文更容易传播错误认知。

## 评价

### 1. 文档设计评价

从站点结构看，这套文档的分层是清晰的：

- `guide`
- `components`
- `advanced`
- `cli`
- `faq`

这套结构符合框架文档的常见阅读路径，也照顾了不同阶段的读者：

- 新用户可以先看指南
- 使用者可以按组件查手册
- 进阶用户可以进入高级章节
- 工具链用户可以直接看 CLI

如果只评价信息架构，本套文档是合格的，甚至可以说是比较完整的。

综合评价：

- 信息架构：`7.5 / 10`

### 2. 写作质量评价

文档整体的语气、解释顺序和章节组织是比较成熟的，尤其是以下方面做得不错：

- 愿意解释设计意图，而不只是列 API
- 有明显的“从概念到实践”的写作顺序
- 组件页通常都有快速开始、配置、示例、最佳实践

但同时也存在比较明显的问题：

- 一些页面保留了明显的 AI 生成痕迹
- 个别段落更像“理想化设计说明”，不是“当前实现说明”
- 部分示例代码没有经过与真实仓库的二次校验

所以如果把它当作“初版框架文档”，写作质量不差；但如果把它当作“当前官方文档”，准确性已经拖累了整体可信度。

综合评价：

- 可读性：`7 / 10`
- 可信度：`4 / 10`

### 3. 与代码一致性评价

这部分是当前最需要正视的问题。

目前不一致不是局部性的，而是分布在多个核心区域：

- Web API 示例
- 路由与中间件说明
- 追踪与日志联动说明
- Redis 包装层示例
- 配置节点名称
- CLI/OpenAPI 标签说明
- 并发章节的整体存在性

这意味着当前文档更像是“某一阶段的设计快照 + 后续补写内容”，而不是严格跟着代码演进的结果。

综合评价：

- 代码一致性：`3 / 10` 到 `4 / 10`

## 结论

### 总体结论

`maltose-docs` 的问题不在“没有结构”，而在“结构已经成型，但内容维护没有跟上代码演进”。

换句话说：

- 这是一套有框架感的文档站
- 但还不是一套可以完全代表当前实现的官方文档

它的优点是：

- 站点组织已经具备完整框架文档的雏形
- 章节覆盖面广
- 写法对新用户比较友好

它的核心短板是：

- 多处公共 API 说明已过期
- 示例代码与当前实现不匹配
- 配置和行为描述存在误导性
- 个别章节描述了并不存在的当前能力

### 更直接的判断

如果把这套文档定位为：

- “内部草稿”或“演进中的框架文档初版”

那么它是有价值的，而且基础不错。

如果把它定位为：

- “当前对外官方文档”

那么还不够稳，特别是在准确性和可复制性上。

### 建议优先级

建议优先处理以下几类内容：

1. 先处理会误导用户写错代码或写错配置的页面
2. 再处理会导致示例无法编译的章节
3. 最后处理概念性偏差和风格统一问题

按模块优先级来看，建议先改：

1. `guide`
2. `components/server`
3. `components/logging`
4. `components/database/mredis`
5. `advanced/full-configuration`
6. `faq`
7. `components/concurrency`

其中 `components/concurrency` 最适合先做一个明确决策：

- 要么补实现
- 要么下线文档
- 要么改成“规划中能力”

如果继续保留为当前能力说明，会持续放大认知偏差。

## 一句话总结

这套文档的站点形态已经像一个正式框架文档，但内容层面仍处于“设计说明多于实现对齐”的阶段；下一步最值得做的，不是继续扩写，而是先按代码把已有内容收紧、校准、去过期。
