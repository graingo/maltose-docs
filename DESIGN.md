# Maltose 文档设计

## 设计目标

- 让第一次接触 Maltose 的开发者在最短路径内运行一个真实接口。
- 让使用者能按组件快速找到公开 API、默认值和边界行为。
- 让生产化主题与基础手册分离，避免单页同时承担教程和参考手册。
- 确保所有可复制示例与当前实现一致。

## 信息架构

| 栏目 | 主要任务 | 不承担的内容 |
| --- | --- | --- |
| Guide | 建立心智模型和完成首次运行 | 完整 API 枚举 |
| Components | 查询组件能力、配置和签名 | 通用部署教程 |
| Advanced | 测试、错误处理、部署和跨边界行为 | 重复组件基础用法 |
| CLI | 命令输入、输出、参数和生成边界 | 业务实现教程 |
| FAQ | 高频故障的短路径诊断 | 大段复制组件手册 |

## 事实来源

出现冲突时按以下顺序校验：

1. 当前 `maltose`、CLI 子模块和 `mconv` 的公开代码与测试。
2. 当前 quickstart 的目录、配置和可运行行为。
3. 文档站已有正文。
4. 历史审计、Issue 和规划说明。

规划中的 API 不应写成当前能力。历史审计文件保留在 `docs/`，但不发布到站点。

## 示例规范

- 示例只使用当前公开符号，字段名和返回值必须与实现一致。
- 省略 import 时，示例必须仍能明确识别所需包；完整示例应可通过 `gofmt`。
- 配置示例区分框架默认值、演示值和生产建议值。
- 涉及外部服务时明确说明 MySQL、Redis、OTLP Collector 等前置条件。
- 不使用不存在的环境变量插值、CLI 参数或隐式初始化行为。
- 对危险操作明确标注作用域，例如 Redis `Clear`、`KEYS` 和数据库迁移。

## 导航与重复内容

- 每个栏目入口页提供按目标选择的阅读路径。
- FAQ 只保留诊断摘要，并链接到唯一的组件或进阶正文。
- 组件间共享的概念放在背景页，具体初始化和 API 留在组件页。
- 站内链接由 VitePress 构建检查，不忽略死链。

## 已知限制

- 文档示例尚未全部进入自动编译测试，仍需在变更时人工对照代码。
- quickstart 和 CLI 可能独立演进，发布前需要同时检查两者。
- 外部系统的部署建议只能作为起点，不能替代容量规划和压测。

## 变更历史

### 2026-07-22 - 全站信息架构优化

**变更内容**：重构栏目入口、FAQ、组件设计哲学和 CLI 阅读路径，启用本地搜索与死链检查。

**变更理由**：此前文档已完成主要实现差异修复，但高频页面仍存在重复、入口不清和设计解释过度的问题。

**影响范围**：Guide、Components、Advanced、CLI、FAQ、VitePress 配置及贡献说明。

**决策依据**：以当前代码、测试和 quickstart 为事实源，按“教程、参考、进阶、排错”分离内容职责。
